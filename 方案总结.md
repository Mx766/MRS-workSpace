# 文档翻译工具 — 调研结论与方案总结

> 2026-06-24，经过两周密集测试后的最终路线

---

## 一、核心结论

### PDF→DOCX 转换：不存在完美工具

PDF 存储字符精确坐标，DOCX 是流式布局，转换本质上有损。经过 7 种工具对比：

| 工具 | 文本型PDF | 图片型PDF | 结论 |
|------|:--:|:--:|------|
| **Acrobat Pro** | ✅ 100%数据完整 | ❌ 乱码 | 文本型首选 |
| **讯飞 PDF OCR** | ❌ 丢数据 | 🟡 英文可读 | 不如Acrobat |
| **Docling** | ✅ 小文件好 | ⚠️ | 大文件OOM |
| **pdf2docx** | 🟡 | ❌ | 格式不可接受 |
| **有道智云** | 🟡 | - | 表格边框丢失 |
| **LibreOffice** | ❌ | ❌ | 表格碎裂 |

### 决定：双轨策略

```
PDF 分类 → 文本型PDF → Acrobat → 后处理(去页眉/修表格) → DeepSeek翻译 → DOCX
         → 图片型PDF → ABBYY FineReader → DeepSeek翻译 → DOCX
```

## 三、工作流

### 标准流程（文本型PDF，占90%）

```
1. Acrobat 打开PDF → 导出为 DOCX ("保留页面布局"模式)
2. 后处理脚本自动:
   - 删除重复页眉/页脚
   - 伪表格(1x1) → 段落或标题
   - Tab拼接 → 正常分段
   - 恢复粗体/字号层级
3. 导入 Gradio 翻译工具
   - 自动加载术语库 → 注入翻译prompt
   - 段落级批量API翻译(保留格式)
   - 进度条实时反馈
4. 输出中文DOCX
5. (可选) 导出术语→Trados XLSX，翻译人员在Trados中校对
```

### 备用流程（图片型PDF，占10%）

```
1. ABBYY FineReader → DOCX
2. 其余同标准流程步骤3-5
```

## 四、已放弃的方向

| 方案 | 放弃原因 |
|------|----------|
| PDF→PDF原位翻译 | 字体/间距/白块问题无法解决，用户判定"没法用" |
| 纯文书翻译 | GLP报告含108张表格，文本型工具处理不了 |
| 讯飞OCR作为主方案 | 丢失关键安全数据(温度/蒸汽压/法规条文号) |
| Docling | 120页GLP报告直接OOM崩溃 |
| pdf2docx | 用户判定"不可接受" |
| FreeP2W | Python 3.14不兼容，无法安装 |
| MinerU | Python 3.14不兼容，等待环境升级 |
