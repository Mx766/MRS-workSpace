# 开发日志 — 2026-06-26

## 1. Issue #12 高亮渲染 — 已解决

**根因**：Word 渲染引擎行为——当 `w:commentReference` 位于段落末尾的独立空 `<w:r>` 中时，Word 抑制同段内所有 `w:shd` + `w:highlight` 的渲染。与 ZIP 重打包、lxml namespace 污染无关。

**修复方案**（v5，`scripts/_build_annotated_v4.py`）：
1. commentReference 放在高亮 run 的 rPr 内部（紧挨 w:shd + w:highlight）
2. 删除 commentRangeStart/End——不需要，还省掉交叉排序 bug
3. lxml 不碰 document.xml——python-docx 处理全部 document.xml 修改 → doc.save() → ZIP 层只加 comments.xml/rels/CT
4. 跨 run 匹配：先单 run 查 target_quote，查不到就拼全文定位起始 run
5. 多长度 key 匹配：[15, 10, 8, 5] 递降，适应 DOCX run 分段

**匹配率**：23/54 (43%) → 51/54 (94%)

## 5. Phase 3/4 字段名不一致导致批注和报告内容为空（下午）

**问题**：Phase 3 机械检查用 `source_value`/`check`/`type`，Phase 4 语义审查用 `source_quote`/`target_quote`/`issue`/`suggestion`/`check_item`。`write_report.py` 和 `write_comments.py` 只读 Phase 4 字段名，导致 96 条机械问题在报告和批注中显示为"原文:（空）译文:（空）建议:（空）"。

**修复**：
- `write_report.py`：新增 `_normalize_issues()` 函数，自动映射 Phase 3 → Phase 4 字段名
- `write_comments.py`：在批注文本构建时 fallback 读取两种字段名
- SKILL.md v2.4：新增 3.1 数据契约，明确两种格式的字段对照表

**教训**：数据生产者（Phase 3/4）和消费者（Phase 6/7）之间必须有明确的字段名契约。脚本改字段名时，所有下游消费端必须同步更新或做好归一化。

## 6. pi=0 机械问题污染报告和批注稿（下午）

**问题**：Phase 3 跨格式（PDF→DOCX）全文匹配产生 96 条 `paragraph_index=0` 问题（如"数字 '03' 缺失"），无法定位到具体段落，均为误报/噪声。但全部 96 条被等量列入报告中等问题清单（占 412 段落）和 DOCX 批注（96 个 bare ref 堆在第一段）。

**修复**：
- `write_report.py`：自动拆分——pi=0 机械问题归入"机械检查参考"汇总行，不逐条列出
- `write_comments.py`：自动跳过 pi=0 机械问题（`issue.get('type') and pi < 1`），不写批注
- SKILL.md v2.4：新增 pi=0 处理规则

**教训**：跨格式对齐的产物（pi=0 = 全文匹配）不可与段落级精准匹配等量齐观。报告用汇总替代逐条，批注直接跳过。

## 7. Phase 4 审查精度差距（下午）

**问题**：本轮主 Agent 直审（v13）仅产出 35 条语义问题，高亮匹配率 49%。对比 v6/v7 深度审查（62 条，匹配率 95%），差距源于：
1. **target_quote 手打而非逐字复制**：加省略号、改符号（℃→C）→ 字符串匹配失败 → bare ref
2. **合并同类问题**：一个段落的多个翻译腔合并为一条"综合建议"，密度缩水 3 倍

**修复**：
- SKILL.md v2.3：新增 4.1.2 精度铁律（A: 逐字复制 / B: 一检查项一 issue）

**教训**：Agent 做语义判断很强，但执行纪律（逐字复制、逐项拆开）需要硬约束。SKILL.md 里的铁律就是用来堵这个漏洞的。

## 9. write_comments.py pi=0 机械问题在批注稿中不可见（下午）

**问题**：v4 的 `write_comments.py` 用 `is_mech = bool(issue.get('type')) and pi < 1` 把 96 条机械检查问题全部 `continue` 跳过。这些问题的 pi=0、有 type 字段，被判定为"跨格式噪声"后既不走段落高亮路径也不走文档级路径 → 校对报告中有汇总，但 DOCX 批注稿中一条都看不到。

**修复**（v5）：
- 移除 `is_mech` 跳过逻辑，pi=0 问题全部进入 `doc_level_issues`
- 段落级问题先分配 comment_id（0..61），文档级后分配（62..157）
- 文档级问题作为 bare ref 插入第一段，批注带 `⚠️ [全文匹配，仅供参考]` 前缀
- 保留 `mech_count` 统计便于 verbose 输出

**验证**：v15 输出 `total_comments=158`（62 para + 96 doc），`highlights=61`，`matched=59`

**教训**：跳过 ≠ 静默丢弃。无法精确定位的问题仍应以文档级批注形式保留（带免责前缀），让译者在 Word 中可见、可逐条复核。

## 10. 机械问题批注格式错乱——"问题"行空白、"建议"行放错误内容

**问题**：`write_comments.py` 对所有 issue 使用统一模板：`原文/X → 译文/Y → 问题/Z → 建议/W`。但机械问题的字段结构不同——`check` 字段存的是完整描述（应为"问题"），却被 `_get(issue, 'suggestion', 'check')` 当作"建议"输出，而真正的 `issue` 字段为空。结果：
```
问题:           ← 空白！
建议: 数字 "03" 在译文中可能缺失...  ← 应该是问题描述
```

**修复**（v5）：
- `is_doc_level` 走专用简化模板：`原文/X → check描述`（无"问题"/"建议"/"译文"标签）
- 段落级保持完整模板不变

**教训**：不同数据源（Phase 3 机械 vs Phase 4 语义）的字段语义不同，不能用同一个模板硬套。doc-level 的 `check` ≠ para-level 的 `suggestion`。
