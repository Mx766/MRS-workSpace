## 1. Issue #12 高亮渲染 — 已解决  `2026-06-26 10:30`

**根因**：Word 渲染引擎行为——当 `w:commentReference` 位于段落末尾的独立空 `<w:r>` 中时，Word 抑制同段内所有 `w:shd` + `w:highlight` 的渲染。与 ZIP 重打包、lxml namespace 污染无关。

**修复方案**（v5，`scripts/_build_annotated_v4.py`）：
1. commentReference 放在高亮 run 的 rPr 内部（紧挨 w:shd + w:highlight）
2. 删除 commentRangeStart/End——不需要，还省掉交叉排序 bug
3. lxml 不碰 document.xml——python-docx 处理全部 document.xml 修改 → doc.save() → ZIP 层只加 comments.xml/rels/CT
4. 跨 run 匹配：先单 run 查 target_quote，查不到就拼全文定位起始 run
5. 多长度 key 匹配：[15, 10, 8, 5] 递降，适应 DOCX run 分段

## 10. 机械问题批注格式错乱 ——"问题"行空白、"建议"行放错误内容  `2026-06-26 16:35`

**问题**：`write_comments.py` 对所有 issue 使用统一模板：`原文/X → 译文/Y → 问题/Z → 建议/W`。但机械问题的字段结构不同——`check` 字段存的是完整描述（应为"问题"），却被 `_get(issue, 'suggestion', 'check')` 当作"建议"输出，而真正的 `issue` 字段为空。结果：
```
问题:           ← 空白！
建议: 数字 "03" 在译文中可能缺失...  ← 应该是问题描述
```

**修复**（v5）：
- `is_doc_level` 走专用简化模板：`原文/X → check描述`（无"问题"/"建议"/"译文"标签）
- 段落级保持完整模板不变

**教训**：不同数据源（Phase 3 机械 vs Phase 4 语义）的字段语义不同，不能用同一个模板硬套。doc-level 的 `check` ≠ para-level 的 `suggestion`。

## 12. 数字/小数检查同样全部假阳性——Phase 3 系统性缺陷  `2026-06-26 17:20`

**问题**：继术语违规 60/60 假阳性后，核实数字/小数 36 条：
- 28 条 `number_missing`：3 条是 DOI/标识符片段、6 条是日期成分（"July 27"）、19 条是参考文献页码（"164–71"）
- 8 条 `decimal_mismatch`：2 条是 DOI 片段、6 条在 DOCX 中实际存在（匹配误差）
- **36/36 = 100% 假阳性**

**结论**：跨格式（PDF→DOCX）场景下，Phase 3 `check_mechanical.py` 的三项主要检查（数字、小数、术语）全部产生 100% 假阳性。根因是脚本对全文做字面匹配，不理解上下文——分不清可翻译数字（测量值）和不可翻译数字（DOI、页码、日期）。

**修复**：
- 新增 `scripts/filter_number_issues.py`：三层过滤（Pattern：DOI/日期/页码 → Presence：在 DOCX 中存在 → 保留人工判断）
- SKILL.md v2.8：新增 [3.3 Phase 3 已知假阳性模式](#33-phase-3-已知假阳性模式必须逐条核实)（四类误报 + Agent 执行规则）
- 确认规则：跨格式场景下，数字/术语检查默认假阳性率 >90%，必须经 filter 脚本清洗后再采信
- `check_mechanical.py` 已有 `>500` 整数过滤（第 487 行），需在 Phase 3 层面加入 Pattern 过滤

**教训**：Phase 3 机械检查在跨格式场景下只是"参考信号"，不是"结论"。同格式（DOCX↔DOCX）逐段对齐的结果才能直接采信。当前 pipeline 必须在 Phase 3 输出后立即运行 `filter_glossary_violations.py` + `filter_number_issues.py`。
