# 开发日志 — 2026-06-26

## 1. Issue #12 高亮渲染 — 已解决

**根因**：Word 渲染引擎行为——当 `w:commentReference` 位于段落末尾的独立空 `<w:r>` 中时，Word 抑制同段内所有 `w:shd` + `w:highlight` 的渲染。与 ZIP 重打包、lxml namespace 污染无关。

**修复方案**（v5，`scripts/_build_annotated_v4.py`）：
1. commentReference 放在高亮 run 的 rPr 内部（紧挨 w:shd + w:highlight）
2. 删除 commentRangeStart/End——不需要，还省掉交叉排序 bug
3. lxml 不碰 document.xml——python-docx 处理全部 document.xml 修改 → doc.save() → ZIP 层只加 comments.xml/rels/CT
4. 跨 run 匹配：先单 run 查 target_quote，查不到就拼全文定位起始 run
5. 多长度 key 匹配：[15, 10, 8, 5] 递降，适应 DOCX run 分段

**匹配率**：23/54 (43%) → 51/54 (94%)

## 3. 术语库修复

**bug**：`load_glossary.py` 列名识别候选列表缺少 `term` 和 `synonym`。术语库 Excel 的实际列名为 `term | description | synonym`，中文翻译在 synonym 列（3044 条），但脚本只识别 `target/zh/chinese` 等列名，导致 fallback 到 col 1（description，几乎全空）。

**修复**：在 src_col 候选列表加 `term`，tgt_col 候选列表加 `synonym`、`translation`。加载量 0 → 2772 条。

**Phase 3 机械检查**：176 条"术语违规"均为 PDF（6 页）vs DOCX（148 段）段落不对齐导致的误报，非翻译质量问题。

## 4. 截断检测误报

**问题**：对比原文和人工 XD 修订版时，用 `python-docx .text` 提取的是未接受修订的文本，导致将正常修订格式误判为"文字截断"。7 条警告中 6 条为误报（如"常见抱怨"→"常见主诉"）。

**教训**：检测修订版文本截断必须用 `w:delText`/`w:ins` XML 级别解析，模拟 accept-all-changes 后的最终文本。
