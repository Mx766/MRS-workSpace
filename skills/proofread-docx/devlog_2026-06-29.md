## 1. WPS 兼容性：批注不可见，仅高亮可见（已完成） `2026-06-29`

**用户反馈**：翻译反应 WPS 打开校对稿只能看见高亮，看不见批注（Word 中正常显示）。

### 1.1 根因

`write_comments.py` 在 6/26 Issue #12 修复中移除了 `w:commentRangeStart`/`w:commentRangeEnd`——当时发现这两个标记 + 段落末尾空 run 中的 `commentReference` 会导致 Word 抑制高亮渲染。修复方案是把 `commentReference` 放进高亮 run 的 `rPr` 内部，去掉 range markers。

**但 WPS 的批注渲染依赖 `w:commentRangeStart`/`w:commentRangeEnd` 来定位批注在文档中的起止位置**。Word 可以仅凭 `commentReference` 推断批注位置，WPS 不行。

核实输出 DOCX：
- `commentReference`: 65 个 ✅
- `commentRangeStart`: **0** ❌
- `commentRangeEnd`: **0** ❌

### 1.2 修复方案

在 `write_comments.py` `write_word_comments()` 中新增 `_add_cmt_range(para_elem, run_elem, cid)` 辅助函数，在每个 `commentReference` 所在 run 的前后插入 `w:commentRangeStart` / `w:commentRangeEnd`。

```xml
<!-- 修复后的 OOXML 结构 -->
<w:commentRangeStart w:id="0"/>    ← WPS 批注起点
<w:r>
  <w:rPr>
    <w:shd .../>                   ← 高亮保留
    <w:highlight .../>
    <w:commentReference w:id="0"/> ← Word 高亮 fix 保留
  </w:rPr>
  <w:t>文本内容</w:t>
</w:r>
<w:commentRangeEnd w:id="0"/>      ← WPS 批注终点
```

**关键**：`commentReference` 仍然在 `rPr` 内部（不在独立空 run 中），因此 Word 的高亮抑制 bug 不会复发。

### 1.3 改动文件

- `skills/proofread-docx/scripts/write_comments.py`：新增 `_add_cmt_range()` 辅助函数 + 5 处调用点：
  1. Strategy 1 匹配成功（已高亮分支）
  2. Strategy 1 匹配成功（新高亮分支）
  3. Strategy 2 跨 run 匹配成功
  4. 段落级 bare ref（未匹配到具体 run）
  5. 文档级 bare ref（pi=0 综合批注）

**教训**：Word 和 WPS 的 OOXML 批注渲染机制不同。Word 凭 `commentReference` 即可定位批注，WPS 需要完整的 `commentRangeStart`/`commentRangeEnd` 标记。移除了 range markers 虽然修好了 Word 高亮，但导致 WPS 完全看不到批注。正确做法是保留 range markers + 把 `commentReference` 放在高亮 run 的 `rPr` 内——两者不矛盾。

## 3. 翻译反馈：不通顺 + 用词不准确 + 跨文件术语统一 `2026-06-29`

**翻译原话**：
> "第一个还是属于不通顺；第二个现在还不属于程序的问题，这个比较特别，当时这批文件有三个，他在另一个文件里面翻的白种人，所以可能涉及跨文件术语统一。最后一个就是用词不准确"

### 3.1 分析

| 问题 | 归属 | 当前能力 |
|------|------|---------|
| **不通顺** | Phase 4 语义审查 | 六维度检查清单有"表达流畅度"但未充分覆盖 |
| **用词不准确** | Phase 4 语义审查 | 有用词准确性维度，但 prompt 可能不够细 |
| **跨文件术语统一** | 新能力 | 当前 pipeline 完全单文件，无跨文件视角 |

### 3.2 根因分析：为什么 AI 漏检了这些问题

**不通顺漏检的 3 个根因**：

| # | 根因 | 说明 |
|---|------|------|
| 1 | **2.19「单读测试」太主观** | AI 读到别扭句时没有强制标记机制。缺少"读到什么就标记"的具体红线信号（如"一句话读两遍才懂→标记"） |
| 2 | **缺少不通顺模式清单** | 翻译腔检查（2.12-2.18）覆盖了 7 种句式级翻译腔，但 **成分残缺**（介词短语当句子）、**句式杂糅**（"目的是为了……"）、**语义冗余**（"进行处理"）这三种高频不通顺模式完全没有对应检查项 |
| 3 | **无逐句强制通读流程** | 当前流程是先看英文再看中文——译者反馈的问题恰恰是"不对比英文才能发现"的不通顺。需要先遮蔽原文通读 |

**用词不准确漏检的 3 个根因**：

| # | 根因 | 说明 |
|---|------|------|
| 1 | **近义词表覆盖不足** | 2.6 只有 8 对近义词，医学翻译中高频混淆组（改善/改进/提升/增强、显示/表明/呈现/证明）未覆盖 |
| 2 | **缺少词义精度检查** | 检查"词对不对"但没有检查"词够不够精确"——AI 判断"改善"译"improve"算对，但实际语境需要"提升"（性能）或"增强"（表达水平） |
| 3 | **缺少语域/语境检查** | 同一段内混用"求美者"和"患者"、抽象名词直译（"X的应用导致……"应为"施加X使……"）等模式无检查项 |

### 3.3 实施方案 ✅

**SKILL.md Phase 4 二、表达规范从 26 项扩充至 34 项**：

1. **新增 2.0 逐句通读强制流程**（检查清单前置步骤）：
   - 4 步流程：遮蔽原文→标记别扭句→反向定位根因→写 issue
   - 7 条红线信号（读到立即标记，不许放过）
   - 每条红线信号关联到具体检查项

2. **新增 2.F 用词精确度增强（4 项）**：
   - 2.F1 词义范围偏差（6 组高频混淆词 + 区分标准）
   - 2.F2 抽象名词过度直译（5 种高频模式 + 判断标准）
   - 2.F3 语域一致性（人称/程度/连接词一致性检查）
   - 2.F4 字面翻译 vs 语境翻译（5 个高频多义词：contact/safety/procedure/control/site + 判断标准）

3. **新增 2.G 不通顺增强（4 项）**：
   - 2.G1 成分残缺（3 种高频模式 + 提取主干检查法）
   - 2.G2 句式杂糅（4 种高频杂糅模式 + 搜索定位法）
   - 2.G3 语义冗余/啰嗦（6 种冗余动词模式 + 删除测试法）
   - 2.G4 逻辑连接断裂（意合断裂 + 话题跳跃检测）

### 3.4 设计原则

- **不破坏原有 26 项编号**：新项目用 2.F/2.G 独立编号
- **每条新检查项附带具体检测方法**：不是"检查一下是否通顺"而是"搜索'进行'+动词，删掉看是否更通顺"

- **每条关联到具体 check_id**：便于追踪和统计分析

## 14. 翻译人员反馈：3 个问题 `2026-06-29 17:30`

### 14.1 反馈原文

> 1. "有的地方说缺符号但是没发现在哪"
> 2. "这个在目录的时候，它提到应该这么修改，但是后文出现这个 test group，它就没再指出来了"
> 3. "还有的时候客户会要求指定词翻译成什么"

### 14.2 问题分析

**问题 1：缺符号定位不清**

根因：机械检查报告 `check: "数字\"4\"在译文中可能缺失"` 只报了缺失值，没有附带 `target_quote` 上下文。译者从报告里看到"缺了数字4"但找不到在段落里的哪个位置。

影响范围：所有 `number_missing` / `decimal_mismatch` 类型问题。

**问题 2：同文档重复问题只报一次**

根因：Phase 4 分批机制中，每批 Agent 独立审查，互不知晓。当 Batch 1（目录区）发现 "test group" 应译为"试验组"，Batch 4-5（正文段落）的 Agent 不知道这个发现，即使后文出现了同样的 "test group" 也不会再标记。

临界场景：
- 同义词替换（如"副反应"→"不良反应"）：第 88-90 段报了 3 条，但如果交错出现在更远的段落就会被跳过
- 术语译法修正（如 "test group" → "试验组"）：TOC 报了，正文里 10+ 处不再报
- 格式问题（如 "kg/m2" → "kg/m²"）：一处报了，其他处忽略

根因本质：Phase 4 只有**逐段检查**，没有**全文档模式传播**。发现一个模式后应该全局扫描。

**问题 3：客户指定术语翻译**

新需求。客户可能有自己的术语偏好表（"这个词必须译成XX"），需要作为强制规则覆盖术语库。当前系统没有加载客户术语要求的入口。

### 14.3 修复方案

**Fix C：机械检查附加上下文定位**（问题 1）
- `check_mechanical.py` 输出 `number_missing`/`decimal_mismatch` 时附带 `target_quote`（从段落文本截取前后 30 字）
- `write_report.py` 报告输出"缺失位置"列，显示缺失数字前后的译文片段

**Fix D：Phase 4.6 全文档模式传播**（问题 2）
- Phase 4 所有批次完成后，新增 4.6 步骤
- 从 `issues_phase4.json` 中提取所有"建议修改"模式（术语替换、格式修正等）
- 对每个模式在 `split_target.json` 中全文正则搜索
- 发现未标记的相同问题 → 自动补报（标注 `source: "pattern_propagation"`）
- Agent 复核补报结果，排除合理差异

**Fix E：客户术语需求加载**（问题 3）
- Phase 0 新增 `client_glossary` 检测：扫描批次目录下的 `_术语要求.xlsx` 或 `client_terms.json`

## 19. v2.17: 弱模型深度引导 + Bug 修复 `2026-07-01`

### 19.1 背景

batch7 DeepSeek Pro 结果分析：

| 指标 | Claude v19 | DS Pro v2.16.1 | 差距 |
|------|-----------|---------------|------|
| issues | 60 | 4 | 15x |
| 覆盖维度 | ~25/34 | 4/50 check_ids | 6x |
| 翻译腔发现 | 大量 2.12-2.18 | 0 | 完全缺失 |
| 用词精度发现 | 大量 2.F1-2.F4 | 0 | 完全缺失 |
| 流畅度发现 | 大量 2.G1-2.G4 | 0 | 完全缺失 |

**根因分析**：

DS Pro 找到了 4 条"表面错误"（重复字 P16、排版截断 P63、单位错误 P120、术语选择 P110），但**完全漏掉了所有需要"中文语感"的维度**——翻译腔、用词精度、句式流畅度、语义冗余。

这不是约束不够紧——v2.16 的校验全绿（0 errors, 0 warnings），说明格式和覆盖率都合规。问题是**模型在执行 34 维度检查时只做了表面扫描，没有深入每个维度的触发条件**。

**关键洞察**：SKILL.md 4.2 的检查清单对 2.C-2.G 维度已经写了详细的触发问题（如"每句问一句：删掉进行/实现/发生/存在/具有后是否更通顺？"），但这些内容在 SKILL.md 里——batch prompt 中只有一句 `[EXISTING 34-ITEM CHECKLIST — see 4.2]`。弱模型不会主动去查阅参考文档，它们只看 prompt 里的内容。

### 19.2 设计方案

**核心思路**：把检查清单的关键触发信息**嵌入 batch prompt 本身**，消除"引用外部文档"的依赖。每个 batch agent 收到的 prompt 是自包含的——不需要去别处查 4.2。

**三大改动**：

#### A. 压缩触发清单嵌入 batch prompt

`prepare_batch_prompt.py` 新增 `--checklist` 模式：读取 SKILL.md 4.2 检查清单，压缩为 ~80 行自包含触发清单，嵌入每个 batch 的 prompt。

每个维度压缩为 1 行：`[check_id] 维度名：触发问题？例：xxx`

对比：
- **v2.16 prompt**: `[EXISTING 34-ITEM CHECKLIST — see 4.2]` → 模型需自行查阅 200+ 行外部文档
- **v2.17 prompt**: 80 行自包含触发清单 → 模型直接在 prompt 中看到每个维度该查什么

#### B. 两轮审查结构

Batch prompt 从"一次性检查全部 34 项"改为两轮：

**第一轮 — 表面错误扫描（快速）**：
- 1.1-1.9 忠实度（漏译/增译/错译/指代/时态等）
- 2.1-2.3 错别字/标点
- 3.1-3.6 数字/单位/符号
- 目标：每段 30 秒，快速扫

**第二轮 — 翻译质量深度审查（仔细）**：
- 2.4-2.11 用词准确性（逐词审视）
- 2.12-2.26 翻译腔/句式/流畅度/MT残留
- 2.F1-2.F4 用词精确度
- 2.G1-2.G4 不通顺增强
- 4.1-4.5 术语合规
- 目标：每段 2-3 分钟，逐句读

强制要求：**第二轮至少找到第一轮 2 倍的问题数**。如果第二轮 issue 数 ≤ 第一轮 → 说明模型只做了表面扫描，第二轮的翻译质量维度被跳过了 → 标记为 incomplete，要求该批次换模型重审。

#### C. Bug 修复

- `calibrate_severity.py`: 修复 `flag_outliers()` 中 `severity` 键被重复赋值覆盖的 bug（line 524 被 line 527 覆盖）

## 23. v2.21: 低频关键术语不遗漏 `2026-07-01`

### 23.1 问题

用户反馈："有的时候出现的不多但也是术语该怎么办"

当前 `extract_key_terminology()` 的硬门槛 `min_occurrences=3`：

```
术语库术语 → 在源文中出现 ≥3 次？──NO──→ 丢弃（不管多重要）
              ↓ YES
           前 30 个高频的 → key_terminology
```

低频但关键的真实场景：

| 场景 | 术语示例 | 次数 | 为什么不能丢 |
|------|---------|------|------------|
| 专利权利要求 | `consisting of` | 1 | open/closed scope 一字之差改变保护范围 |
| SDS | `H319` | 1 | 全文唯一的危害说明，法定译法不可改 |
| 渗透测试报告 | `CVE-2024-12345` | 1 | 漏洞编号译错就引用不到 |
| 客户要求 | `handpiece → 手持件` | 2 | 客户明确要求，不检查就是事故 |
| 论文标题 | `neocollagenesis` | 1 | 标题里的核心概念，全文关键 |

**附加发现**：`--client-glossary` 参数在 `inject_context.py` 中是死代码——`ArgumentParser` 声明了但 `main()` 从未读取、从未传给 `extract_key_terminology()`。

### 23.2 设计方案

**核心思路**：从"一刀切频率过滤"变为"多层信号加权"——低频但重要的术语也有通道进入 Phase 4 审查。

```
                    术语库术语 + 客户术语
                        │
        ┌───────────────┼─────────────────┐
        │               │                 │
   客户要求术语      出现在标题/      出现在正文
   (client mandated)  表格/首段           │
        │               │                 │
   无条件保留         count≥1 保留     count≥3 保留
   importance:        importance:      importance:
   client_mandated    structural       frequency
        │               │                 │
        └───────────────┴─────────────────┘
                        │
            critical_low_freq_terms    key_terminology
            (新增输出字段)             (现有字段，不变)
```

**三路信号**：

| 信号 | 判断方式 | 阈值 |
|------|---------|------|
| **客户要求** (`client_mandated`) | 术语 key 匹配客户 `_术语要求.xlsx` | 出现 ≥1 次即保留，无视最短长度限制 |
| **结构位置** (`structural`) | 术语出现在标题/表格/首 3 段 | 出现 ≥1 次即保留 |
| **领域关键** (`domain_critical`) | 匹配关键标识符模式（H-code/CVE/CAS/专利号等） | 出现 ≥1 次即保留 |

结构化位置检测复用已有数据：
- `source_paras[].heading_level` → 标题
- `source_paras[].style == "Table"` → 表格
- `source_paras[:3]` → 摘要/引言区域

领域关键模式：
```
H\d{3}[A-Z]?       GHS 危害/防范说明代码
CVE-\d{4}-\d+       CVE 漏洞编号
[A-Z]{2,4}\s*\d+    ISO/ASTM/EN 标准编号  
\d{2,7}-\d{2}-\d    CAS 号
NCT\d{8}            临床试验注册号
```

### 23.3 涉及文件

| 文件 | 改动 |
|------|------|
| `scripts/inject_context.py` | `extract_key_terminology()`: 加 `client_glossary`/`source_paras` 参数；三路信号判断；返回 `(key_terminology, critical_low_freq_terms)` 二元组；`main()`: 修死代码，加载 client_glossary，传 source_paras，输出新字段 |
| `scripts/prepare_batch_prompt.py` | `build_batch_prompt_context()`: 读取 `critical_low_freq_terms`，合并进 batch prompt 的 term 上下文，标注重要性原因 |

| `devlog_2026-06-29.md` | Section 23（本文） |

## 25. v2.23: Pipeline 对齐修复 — 补桥 + 强化匹配 `2026-07-02`

### 25.1 背景

### 25.2 改动一：新建 `build_batches.py`

**路径**：`scripts/build_batches.py`（全新，~180行）

**职责**：连接 `split_target.json`/`split_source.json` 和 `prepare_batch_prompt.py` 的桥梁。

**功能**：
- 从 split JSON 读取所有段落（index 对齐 doc.paragraphs）
- 从 split JSON 读取所有表格单元格，按 chapter 段落范围分组
- 按 15 段一批切分段落，表格按负载均衡分配到覆盖的 batch
- 可选注入 phase4_context.json 的高风险标记
- 输出 `batch_NN_data.json`（与 `prepare_batch_prompt.py` 期望格式兼容）

**关键函数**：
- `collect_paragraph_entries()` — 收集段落（index = split 全局索引，对齐 doc.paragraphs）
- `collect_table_entries()` — 收集表格单元格（index 从 total_paras+1 递增，对齐 write_comments.py 的 `_build_table_cell_map()`）
- `split_paras_into_batches()` — 段落分批
- `attach_tables_to_batches()` — 表格负载均衡分配（候选 batch 中选 tables 最少的）

**测试结果**（rhBMP-2 文档）：
```
124 batches, 1856 paragraphs + 103 tables (16,205 cells) (batch_size=15)
```
表格从原来的全集中在 2 个 batch，改为按章负载均衡分布。

### 25.3 改动二：强化 `write_comments.py`

**路径**：`scripts/write_comments.py`

**三处增强**：

**a) Strategy 3 fallback 模糊匹配**：
- 3a: 去空白后精确匹配 → 处理 DOCX 中空白符差异导致的匹配失败
- 3b: 最长中文片段匹配（≥4字）→ 处理 target_quote 只剩中文核心片段的场景

**b) 邻段搜索（±5段）**：
- 当本段为空或 target 不在本段时，自动搜索上下 5 段
- 找到匹配则在该邻段添加高亮+批注
- 大幅减少"段落为空"和"target不在本段"导致的裸引

**c) 匹配失败诊断输出**：
```
[write_comments] 匹配统计: exact=20, fuzzy=5, neighbor=2, bare=279
[write_comments] 匹配率: 27/306 = 9%
[write_comments] ⚠ 匹配率仅9%——建议检查paragraph_index是否与DOCX对齐
[write_comments] 失败原因: empty_target=1, para_empty=187, target_not_found=91
```
匹配率 <50% 时自动输出告警，指引 Agent 检查 paragraph_index 对齐和 target_quote 逐字复制。
