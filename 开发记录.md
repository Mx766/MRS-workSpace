### 5. 讯飞 OCR 测试

#### 5.1 OCRforLLM (通用文档识别/se75ocrbm)

| 尝试 | 结果 | 问题 |
|------|------|------|
| 鉴权测试 | ✅ 通过 | HMAC-SHA256签名，host=cn-huabei-1.xf-yun.com |
| 真实OCR | ❌ licc failed | 错误码11201，服务未授权/未开通 |

**鉴权方式**：标准讯飞HMAC-SHA256（URL参数 + Date Header）

#### 5.2 PDF文档识别 (pdfOcr, iocr.xfyun.cn)

| 尝试 | 结果 | 问题 |
|------|------|------|
| 鉴权测试 | ✅ 通过 | **不同鉴权**: MD5(appId+timestamp) → HMAC-SHA1 → Base64 |
| MSDS 8页测试 | ✅ 返回Word | 66KB DOCX，直接传PDF→返回Word文件 |

**鉴权方式**（与标准不同！）：
1. `auth = MD5(appId + timestamp)` → 32位小写hex
2. `signature = Base64(HMAC-SHA1(auth, apiSecret))`
3. 请求头: `appId`, `timestamp`(Unix秒), `signature`

**API端点**：
- 上传: `POST https://iocr.xfyun.cn/ocrzdq/v1/pdfOcr/start` (multipart/form-data)
- 查状态: `GET https://iocr.xfyun.cn/ocrzdq/v1/pdfOcr/status?taskNo=xxx`
- 完成后返回 `downUrl` 下载Word文件

### 6. MSDS 三版对比 (原始PDF vs Acrobat vs 讯飞OCR)

对比文件: `CH2.4.1(4-2) 1. MSDS_Barrel_TOPAS MSDS-EN V8.00_0 (2).pdf` (8页MSDS)

#### 6.1 数据完整性

| 关键数据 | 原始PDF | Acrobat | 讯飞OCR |
|----------|---------|---------|----------|
| `TOPAS®` | ✅ | ✅ | ❌ → `TOPASCyclic` |
| `§1910.1200` | ✅ | ✅ | ❌ → 消失/`S1910.1200` |
| `350 °C / 662F` | ✅ | ✅ | ❌ 温度完全丢失 |
| `< 0.001 mm Hg @25°C` | ✅ | ✅ | ❌ 只剩 `mm Hg @` |
| `VOC < 0.5 %` | ✅ | ✅ | ❌ 表格空白 |
| `***` 变更标记 | ✅ | ✅ | ❌ 全部丢失 |
| `CAS 26007-43-2` | ✅ | ✅ | ✅ |

**结论**：Acrobat 数据完整性 100%，讯飞 OCR 丢失关键安全数据，不可用于专业文档。

#### 6.2 结构问题 (Acrobat)

| 问题 | 严重度 | 详情 |
|------|--------|------|
| 页眉/页脚污染 | 🔴 | `TOPAS® Cyclic Olefin Copolymers` 在正文重复12次，Revision Date/product code 重复7次 |
| 节标题→表格 | 🔴 | 16个标题(`1: Identification`~`16: Other`)全变成1×1表格 |
| 页眉→表格 | 🔴 | 7个header表格 + 6个页码表格 |
| Tab拼接 | 🟡 | 4处用`\t`把不同字段硬拼同段 |
| 格式丢失 | 🟡 | 7种字号(8~20pt)→全Normal，粗体→0 |
| 伪表格 | 🟡 | 36表中27个是假的，仅9个真表格 |

#### 6.3 结论

**Acrobat 远优于讯飞 OCR。** Acrobat 问题都是结构性的（页眉重复、伪表格、格式丢失），可脚本修复。讯飞 OCR 是数据丢失，不可接受。

**推荐路线**：Acrobat PDF→DOCX → 脚本后处理(去页眉/修表格/恢复格式) → DeepSeek翻译 → DOCX输出

### 9. PyMuPDF 段落级翻译（translate_paragraph.py）

**目标**：不解构 PDF，直接在原位替换文字 + 翻译

**关键技术**：
- **段落分组**：按 y 间距（阈值 6pt）+ x 重叠度将行合并为段落
- **背景擦除**：采样文本周围背景色 → `draw_rect` 填充 → 覆盖原文
- **字体嵌入**：`C:\Windows\Fonts\simhei.ttf`（黑体），解决 `china-ss` 不渲染问题
- **API 翻译**：DeepSeek Anthropic-compatible endpoint，30 段/批，`<<<SEP>>>` 分隔
- **缓存**：MD5 前 12 位 hex 作为 key，JSON 持久化到 `cache/`

**字体 Bug 修复**：
- ❌ `_CN_FONT = page.insert_font(...)` → 返回 int，后续 `insert_textbox(fontname=int)` 崩溃
- ✅ `get_cn_font()` 始终返回字符串 `'SimHei'`

### 10. 有道智云 PDF→DOCX API 测试

**API 信息**：
- 文档：`https://ai.youdao.com/DOCSIRMA/html/ocr/api/pdf-wd-premium/`
- 端点：`https://openapi.youdao.com/file_convert/v2`
- 流程：`POST /upload`（multipart/form-data）→ `POST /query` 轮询 → 下载
- 签名：`SHA256(appKey + input + salt + curtime + appSecret)`，v3
- 价格：0.15 元/页
- 凭证：appKey=`0576c0f9f7b44770`, appSecret=`xPh1pkMTt7R1ckzlXPRRnG91xpek3mJI`

**踩坑记录**：
| 问题 | 原因 | 解决 |
|------|------|------|
| 503 未知错误 | httpx `data=` 是 form-encoded 而非 multipart | 手动构建 multipart body + UUID boundary |
| 签名 salt 不一致 | `make_sign()` 每次生成新 salt，但请求体和签名用了不同值 | `make_sign()` 增加 `salt`/`curtime` 可选参数 |

### 16. Word COM PDF 导入（扫描件 OCR）

**发现**：Word 自带 PDF 导入引擎对扫描件有内置 OCR，效果远超预期。

**测试文档**：`Attachment2. Microbiologica.pdf`（32页纯扫描件，0内嵌文字，410张图）

**方法**：Python + `pywin32` COM 自动化

```python
import win32com.client, pythoncom
pythoncom.CoInitialize()
word = win32com.client.Dispatch('Word.Application')
word.Visible = False
word.DisplayAlerts = 0  # 抑制对话框
word.AutomationSecurity = 3  # 禁用宏

doc = word.Documents.Open(pdf_path, ConfirmConversions=False,
    ReadOnly=True, Format='PDF Files', NoEncodingDialog=True)
doc.SaveAs2(output_path, FileFormat=16)  # wdFormatDocumentDefault
doc.Close()
word.Quit()
pythoncom.CoUninitialize()
```

**结果**：

| 指标 | 数值 |
|------|------|
| 输出大小 | 6,666 KB |
| 段落 | 239（150正文 + 15标题 + 10页码 + 64空） |
| 表格 | **25 个** |
| 文字量 | 12,408 字符 |
| OCR 质量 | 良好（少量日期双点号、字母间距异常） |

**关键配置**：
- `DisplayAlerts=0` — 必须，否则 Word 会弹对话框导致 COM 调用挂起
- `Format='PDF Files'` — 触发 Word 内置 PDF 导入引擎（含 OCR）
- `FileFormat=16` — DOCX 格式
- 需安装 `pywin32`（`pip install pywin32`）

**对比**：

| 方案 | 扫描件效果 | 复杂度 |
|------|:--:|:--:|
| Word COM | ✅ 12K字+25表 | 3行代码 |
| 讯飞 PDF OCR | ❌ 丢数据 | API对接 |
| Acrobat | ❌ 乱码 | 手动操作 |

**结论**：扫描件 PDF → Word COM 导入 → DOCX，简单且效果最好。之前 PowerShell COM 调用失败是因为 bash 吞了 `$` 变量符号，改用 Python `win32com` 解决。

**脚本**：`scripts/word_com_ocr.py`
